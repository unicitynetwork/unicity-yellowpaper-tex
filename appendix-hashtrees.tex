\chapter{Hash Trees}\label{app:hashtrees}

\section{Plain Hash Trees}

\subsection{Function $\textsc{plain\_tree\_root}$}\label{sec:plain-tree-root}

Computes the root value of the plain hash tree with the given $n$ values in its leaves.

\textbf{Input}:
$L = \langle x_1, \ldots, x_n \rangle \in \hashtype^n$, the list of the values in the $n$ leaves of the tree

\textbf{Output}:
$r \in \hashtype\opt$, the value in the root of the tree

\textbf{Computation}:
\begin{algorithmic}
	\Function{plain\_tree\_root}{$L$}
		\If {$n = 0$} \Comment{$L = \langle \rangle$}
			\State \Return $\bot$
		\ElsIf {$n = 1$} \Comment{$L = \langle x_1 \rangle$}
			\State \Return $x_1$
		\Else
			\State $m \gets 2^{\lfloor \log_2(n-1) \rfloor}$ \Comment{Canonical tree}
			\State $L_\mathsf{left} \gets \langle x_1, \ldots, x_m \rangle$
			\State $L_\mathsf{right} \gets \langle x_{m+1}, \ldots, x_n \rangle$
			\State \Return $H(\Call{plain\_tree\_root}{L_\mathsf{left}} , \Call{plain\_tree\_root}{L_\mathsf{right}})$
		\EndIf
	\EndFunction
\end{algorithmic}

Note that $2^{\lfloor \log_2(n-1) \rfloor}$ is the value of the highest $1$-bit in the binary representation of $n - 1$, which may be the preferred way to compute $m$ in some environments. Splitting the leaves this way results in a structure that allows the root of the tree to be computed incrementally, without having all the leaves in memory at once.

\subsection{Function $\textsc{plain\_tree\_chain}$}\label{sec:plain-tree-chain}

Computes the hash chain from the $i$-th leaf to the root of the plain hash tree with the given $n$ values in its leaves.

\textbf{Input}:
\begin{enumerate}
	\item $L = \langle x_1, \ldots, x_n \rangle \in \hashtype^n$, the list of the values in the $n$ leaves of the tree
	\item $i \in \{1, \ldots, n\}$, the index of the starting leaf of the chain
\end{enumerate}

\textbf{Output}:
$C = \langle (b_1, y_1), \ldots, (b_\ell, y_\ell) \rangle \in (\bool \times \hashtype)^\ell$, where $y_j$ are the sibling hash values on the path from the $i$-th leaf to the root and $b_j$ indicate whether the corresponding $y_j$ is the right- or left-hand sibling

\textbf{Computation}:
\begin{algorithmic}
	\Function{plain\_tree\_chain}{$L; i$}
		\State \Assert $1 \le i \le n$
		\If {$n = 1$} \Comment{$L = \langle x_1 \rangle$}
			\State \Return $\langle \rangle$
		\Else
			\State $m \gets 2^{\lfloor \log_2(n-1) \rfloor}$ \Comment{Must match \textsc{plain\_tree\_root}}
			\State $L_\mathsf{left} \gets \langle x_1, \ldots, x_m \rangle$
			\State $L_\mathsf{right} \gets \langle x_{m+1}, \ldots, x_n \rangle$
			\If {$i \le m$}
				\State \Return $\Call{plain\_tree\_chain}{L_\mathsf{left}; i} \| (0, \Call{plain\_tree\_root}{L_\mathsf{right}})$
			\Else
				\State \Return $\Call{plain\_tree\_chain}{L_\mathsf{right}; i-m} \| (1, \Call{plain\_tree\_root}{L_\mathsf{left}})$
			\EndIf
		\EndIf
	\EndFunction
\end{algorithmic}

\subsection{Function $\textsc{plain\_tree\_output}$}\label{sec:plain-tree-output}

Computes the output hash of the chain $C$ on the input $x$.

\textbf{Input}:
\begin{enumerate}
	\item $C = \langle (b_1, y_1), \ldots, (b_\ell, y_\ell) \rangle \in (\bool \times \hashtype)^\ell$, where $y_j$ are the sibling hash values on the path from the $i$-th leaf to the root and $b_j$ indicate whether the corresponding $y_j$ is the right- or left-hand sibling
	\item $x \in \hashtype$, the input hash value
\end{enumerate}

\textbf{Output}:
$r \in \hashtype$, output value of the hash chain

\textbf{Computation}:
\begin{algorithmic}
	\Function{plain\_tree\_output}{$C; x$}
		\If {$\ell = 0$} \Comment{$C = \langle \rangle$}
			\State \Return $x$
		\Else
			\State \Assert $b_\ell \in \bool$
			\If {$b_\ell = 0$}
				\State \Return $H(\Call{plain\_tree\_output}{\langle (b_1, y_1), \ldots, (b_{\ell-1}, y_{\ell-1}) \rangle; x}, y_\ell)$
			\Else
				\State \Return $H(y_\ell, \Call{plain\_tree\_output}{\langle (b_1, y_1), \ldots, (b_{\ell-1}, y_{\ell-1}) \rangle; x})$
			\EndIf
		\EndIf
	\EndFunction
\end{algorithmic}

\subsection{Inclusion Proofs}

Plain hash trees can be used to provide and verify inclusion proofs. The process for this is as follows:

\begin{itemize}
	\item To commit to the contents of a list $L = \langle x_1, \ldots, x_n \rangle$:
		\begin{itemize}
			\item Compute $r \gets \Call{plain\_tree\_root}{L}$.
			\item Authenticate $r$ somehow (sign it, post it to an immutable ledger, etc).
		\end{itemize}
	\item To generate inclusion proof for $x_i \in L$:
		\begin{itemize}
			\item Compute $C \gets \Call{plain\_tree\_chain}{L; i}$.
		\end{itemize}
	\item To verify the inclusion proof $C = \langle (b_1, y_1), \ldots, (b_\ell, y_\ell) \rangle$ for $x$:
		\begin{itemize}
			\item Check that $\Call{plain\_tree\_output}{C; x} = r$, where $r$ is the previously authenticated root hash value.
		\end{itemize}
\end{itemize}

\section{Indexed Hash Trees}

\subsection{Function $\textsc{index\_tree\_root}$}\label{sec:index-tree-root}

Computes the root value of the indexed hash tree with the given $n$ key-value pairs in its leaves.

\begin{figure}[h]
	\begin{center}
		\includegraphics[height=5cm]{pic/index-tree.png}
		\caption{Keys of the nodes of an indexed hash tree.}\label{fi:index-tree}
	\end{center}
\end{figure}

\textbf{Input}:
List $L = \langle (k_1, x_1), \ldots, (k_n, x_n) \rangle \in (\mathbb{K} \times \hashtype)^n$, the list of the key-value pairs in the $n$ leaves of the tree; $\mathbb{K}$ must be a linearly ordered type and the input pairs must be strictly sorted in this order, i.e. $k_1 < \ldots < k_n$

\textbf{Output}:
$r \in \hashtype\opt$, the value in the root of the tree

\textbf{Computation}:

\begin{algorithmic}
	\Function{index\_tree\_root}{$L$}
		\State \Assert $k_1 < \ldots < k_n$
		\If {$n = 0$} \Comment{$L = \langle \rangle$}
			\State \Return $\bot$
		\ElsIf {$n = 1$} \Comment{$L = \langle (k_1, x_1) \rangle$}
			\State \Return $H(1, k_1, x_1)$
		\Else
			\State $m \gets \lceil n/2 \rceil$ \Comment{Most balanced tree}
			\State $L_\mathsf{left} \gets \langle(k_1, x_1), \ldots, (k_m, x_m) \rangle$
			\State $L_\mathsf{right} \gets \langle (k_{m+1}, x_{m+1}), \ldots, (k_n, x_n) \rangle$
			\State \Return $H(0, k_m, \Call{index\_tree\_root}{L_\mathsf{left}}, \Call{index\_tree\_root}{L_\mathsf{right}})$
		\EndIf
	\EndFunction
\end{algorithmic}

\subsection{Function $\textsc{index\_tree\_chain}$}\label{sec:index-tree-chain}

Considers the indexed hash tree with the given $n$ key-value pairs in its leaves. If there is a leaf containing the key $k$, computes the hash chain from that leaf to the root. If there is no such leaf, computes the hash chain from the leaf where $k$ should be according to the ordering, which can be used as a proof of $k$'s absence.

\textbf{Input}:
\begin{enumerate}
	\item $L = \langle (k_1, x_1), \ldots, (k_n, x_n) \rangle \in (\mathbb{K} \times \hashtype)^n$, the list of the key-value pairs in the $n$ leaves of the tree; $\mathbb{K}$ must be a linearly ordered type and the input pairs must be strictly sorted in this order, i.e. $k_1 < \ldots < k_n$
	\item $k \in \mathbb{K}$, the key to compute the path for
\end{enumerate}

\textbf{Output}:
$C = \langle (k_1, y_1), \ldots, (k_\ell, y_\ell) \rangle \in (\mathbb{K} \times \hashtype)^\ell$, where $k_j$ are the keys in the nodes on the path from the leaf to the root and $y_j$ are the sibling hash values

\textbf{Computation}:
\begin{algorithmic}
	\Function{index\_tree\_chain}{$L; k$}
		\State \Assert $k_1 < \ldots < k_n$
		\If {$n \in \{0,1\}$} \Comment{$L = \langle \rangle$ or $L = \langle (k_1, x_1) \rangle$}
			\State \Return $L$
		\Else
			\State $m \gets \lceil n/2 \rceil$ \Comment{Must match \textsc{index\_tree\_root}}
			\State $L_\mathsf{left} \gets \langle(k_1, x_1), \ldots, (k_m, x_m) \rangle$
			\State $L_\mathsf{right} \gets \langle (k_{m+1}, x_{m+1}), \ldots, (k_n, x_n) \rangle$
			\If {$k \le k_m$}
				\State \Return $\Call{index\_tree\_chain}{L_\mathsf{left}; k} \| (k_m, \Call{index\_tree\_root}{L_\mathsf{right}})$
			\Else
				\State \Return $\Call{index\_tree\_chain}{L_\mathsf{right}; k} \| (k_m, \Call{index\_tree\_root}{L_\mathsf{left}})$
			\EndIf
		\EndIf
	\EndFunction
\end{algorithmic}

\subsection{Function $\textsc{index\_tree\_output}$}\label{sec:index-tree-output}

Computes the output hash of the chain $C$ on the input key $k$.

\textbf{Input}:
\begin{enumerate}
	\item $C = \langle (k_1, y_1), \ldots, (k_\ell, y_\ell) \rangle \in (\mathbb{K} \times \hashtype)^\ell$, where $k_j$ are the keys in the nodes on the path from the leaf to the root and $y_j$ are the sibling hash values
	\item $k \in \mathbb{K}$, the input key
\end{enumerate}

\textbf{Output}:
$r \in \hashtype\opt$, the value in the root of the tree

\textbf{Computation}:
\begin{algorithmic}
	\Function{index\_tree\_output}{$C; k$}
	\If {$\ell = 0$} \Comment{$C = \langle \rangle$}
		\State \Return $\bot$
	\ElsIf {$\ell = 1$} \Comment{$C = \langle (k_1, y_1) \rangle$}
		\State \Return $H(1, k_1, y_1)$
	\Else
		\If {$k \le k_\ell$}
			\State \Return $H(0, k_\ell, \Call{index\_tree\_output}{\langle (k_1, y_1), \ldots, (k_{\ell-1}, y_{\ell-1}) \rangle; k}, y_\ell)$
		\Else
			\State \Return $H(0, k_\ell, y_\ell, \Call{index\_tree\_output}{\langle (k_1, y_1), \ldots, (k_{\ell-1}, y_{\ell-1}) \rangle; k})$
		\EndIf
	\EndIf
	\EndFunction
\end{algorithmic}

\subsection{Inclusion and Exclusion Proofs}

Indexed hash trees can be used to provide and verify both inclusion and exclusion proofs. The process for this is as follows:

\begin{itemize}
	\item To commit to the contents of a list $L = \langle (k_1, x_1), \ldots, (k_n, x_n) \rangle$ (where $k_1 < \ldots < k_n$):
		\begin{itemize}
			\item Compute $r \gets \Call{index\_tree\_root}{L}$.
			\item Authenticate $r$ somehow (sign it, post it to an immutable ledger, etc).
		\end{itemize}
	\item To generate inclusion proof for $(k_i, x_i) \in L$:
		\begin{itemize}
			\item Compute $C \gets \Call{index\_tree\_chain}{L; k_i}$.
		\end{itemize}
	\item To verify the inclusion proof $C = \langle (k_1, y_1), \ldots, (k_\ell, y_\ell) \rangle$ for $(k, x)$:
		\begin{itemize}
			\item Check that $\Call{index\_tree\_output}{C; k} = r$, where $r$ is the previously authenticated root hash value.
			\item Check that $(k, x) = (k_1, y_1)$, where $(k_1, y_1)$ is the first pair in the list $C$.
		\end{itemize}
	\item To generate exclusion proof for $k \not\in \{k_1, \ldots, k_n\}$:
		\begin{itemize}
			\item Compute $C \gets \Call{index\_tree\_chain}{L; k}$.
		\end{itemize}
	\item To verify the exclusion proof $C = \langle (k_1, y_1), \ldots, (k_\ell, y_\ell) \rangle$ for $k$:
		\begin{itemize}
			\item Check that $\Call{index\_tree\_output}{C; k} = r$, where $r$ is the previously authenticated root hash value.
			\item Check that $k \ne k_1$, where $(k_1, y_1)$ is the first pair in the list $C$.
		\end{itemize}
\end{itemize}


\section{Radix Sparse Merkle Trees}\label{app:rsmt}

A \emph{Radix Sparse Merkle Tree}\index{Radix Sparse Merkle Tree} (RSMT) is a leaf-anchored, path-compressed binary Merkle tree over a 256-bit key space. It authenticates a finite map $k \mapsto v$, where every stored key appears as exactly one leaf and every internal node records the absolute bit position at which its descendant keys bifurcate, together with the key prefix (the \emph{region}) shared by all keys below it. The implementation documented here uses byte-order-preserving LSB-in-byte bit numbering. Each leaf hash commits to the full 256-bit key, and each internal node hash commits to the explicit bifurcation depth and to the region. Depth and region are absolute properties of a node---splitting an edge above an unchanged subtree alters neither---so batch insertion rehashes only the changed search frontier and the newly created nodes. The region commitment pins every node to its position in the key space; the consistency proof (Sec.~\ref{sec:rsmt-consistency-proof}) relies on it to verify that every new leaf is placed at the position determined by its own key. The tree supports inclusion certificates, non-inclusion certificates and consistency proofs.

\subsection{Tree Structure}

Keys are elements of $\bytes{32}$ (256 bits). Write $k=k_0\|k_1\|\cdots\|k_{31}$ in wire byte order. Bit position $i$ addresses bit $i\bmod 8$ of byte $k_{\lfloor i/8\rfloor}$, where bit $0$ of a byte is its least significant bit. Thus depth $0$ reads the low bit of $k_0$, depth $7$ reads the high bit of $k_0$, depth $8$ reads the low bit of $k_1$, and depth $255$ reads the high bit of $k_{31}$. Tree traversal MUST NOT reinterpret the key as a single big-endian integer for bit addressing.

\subsubsection{Node Format}

For a key $k$ and a depth $d$, we write $k[d]$ for the bit of $k$ at position $d$ and $k[0..d)$ for the bit string formed by the bits of $k$ at positions $0, \ldots, d-1$, all in the bit numbering above.

\begin{itemize}
	\item \textbf{Empty tree}: The empty tree has root $\bot$.
	\item \textbf{Leaf node}: A leaf stores a pair $(k, v)$ with $k \in \bytes{32}$ and $v \in \bytestr$. The full key $k$ anchors the leaf position in the tree. A leaf's region is its full key and its depth is $256$.
	\item \textbf{Internal node}: An internal node (\emph{junction}) stores a tuple $(d, p, c_0, c_1)$, where $d \in \{0, \ldots, 255\}$ is the bifurcation depth, $p \in \bits{d}$ is the \emph{region}: the common value $k[0..d)$ of all descendant keys $k$, and $c_0$ and $c_1$ are the left and right child subtrees. A search key $k$ may descend through the node if and only if $k[0..d) = p$; the next bit $k[d]$ then selects child $c_0$ for $0$ and child $c_1$ for $1$.
\end{itemize}

The region both routes lookups and is part of the Merkle commitment: the commitment is determined by the leaf contents, the child hashes, the absolute bifurcation depths and the regions.

\subsection{Hash Computation}

The hash function $H$ is SHA-256. Domain separation is achieved via a one-byte prefix: $\mathtt{0x00}$ for leaves and $\mathtt{0x01}$ for internal nodes.

For hashing, a region $p \in \bits{d}$ is packed into a 32-byte string $\langle p \rangle$: the bits of $p$ are placed at bit positions $0, \ldots, d-1$ (in the bit numbering of this appendix) and all bits at positions $d, \ldots, 255$ are zero. Equivalently, $\langle p \rangle$ is obtained from any descendant key by clearing all bits at positions $\ge d$. Together with the one-byte depth, the packing is injective.

\subsubsection{Function $\textsc{rsmt\_leaf\_hash}$}\label{sec:rsmt-leaf-hash}

Computes the hash of an RSMT leaf node.

\textbf{Input}:
\begin{enumerate}
	\item $k \in \bytes{32}$ -- key (256-bit identifier)
	\item $v \in \bytestr$ -- value
\end{enumerate}

\textbf{Output}: $h \in \hashtype$

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_leaf\_hash}{$k, v$}
		\State \Return $H(\mathtt{0x00} \,\|\, k \,\|\, v)$
	\EndFunction
\end{algorithmic}

\subsubsection{Function $\textsc{rsmt\_node\_hash}$}\label{sec:rsmt-node-hash}

Computes the hash of an RSMT internal node.

\textbf{Input}:
\begin{enumerate}
	\item $h_L \in \hashtype$ -- left child hash
	\item $h_R \in \hashtype$ -- right child hash
	\item $d \in \uint{8}$ -- bifurcation depth
	\item $p \in \bits{d}$ -- region
\end{enumerate}

\textbf{Output}: $h \in \hashtype$

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_node\_hash}{$h_L, h_R, d, p$}
		\State \Return $H(\mathtt{0x01} \,\|\, d \,\|\, \langle p \rangle \,\|\, h_L \,\|\, h_R)$
	\EndFunction
\end{algorithmic}

Every leaf hash commits to the full key, and every internal-node hash commits to the absolute branching depth $d$ and the absolute region $p$. Splitting an edge above a node changes neither its depth nor its region, so an insertion never recomputes any pre-existing node hash. Inclusion certificates do not transmit regions: the verifier derives each expected region from the queried key itself (Sec.~\ref{sec:rsmt-verify-inclusion}).

\subsection{Batch Insertion}\label{sec:rsmt-batch-insert}

The batch insertion algorithm sorts the input batch once in the traversal order defined in Section~\ref{sec:rsmt-sort-order}, removes repeated keys within the batch, and then merges the resulting sorted list with the affected frontier of the existing tree. Existing keys are dropped from the batch during this merge.

\subsubsection{Function $\textsc{rsmt\_batch\_insert\_np}$}\label{sec:rsmt-batch-insert-np}

In the following, the helper procedure $\textsc{rsmt\_merge\_batch}$ recursively merges a contiguous slice of the sorted batch into the current subtree root, and $\textsc{rsmt\_build\_subtree}$ builds a new radix subtree from a non-empty sorted slice.

\textbf{Input}:
\begin{enumerate}
	\item $T$ -- current RSMT
	\item $B = \langle (k_1, v_1), \ldots, (k_m, v_m) \rangle$ -- unordered batch of candidate insertions
\end{enumerate}

\textbf{Output}:
$T$ updated in place, with root replaced by the root of the post-state tree

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_batch\_insert\_np}{$T, B$}
		\State sort $B$ by $\textsc{rsmt\_sort\_key}$ and remove repeated keys to obtain $B'$
		\If{$B' = \langle \rangle$}
			\State \Return
		\EndIf
		\State $T.\mathsf{root} \gets \Call{rsmt\_merge\_batch}{T.\mathsf{root}, B'}$
	\EndFunction
\end{algorithmic}


The helper procedure $\textsc{rsmt\_merge\_batch}(r, S)$ receives a current subtree root $r$ and a contiguous slice $S$ of the traversal-sorted batch. Because depths and regions are absolute, no traversal position needs to be threaded through the recursion. The following pseudocode omits low-level bit manipulations.

\begin{algorithmic}
	\Function{rsmt\_merge\_batch}{$r, S$}
		\If{$S = \langle \rangle$}
			\State \Return $r$
		\EndIf
		\If{$r = \bot$}
			\State \Return \Call{rsmt\_build\_subtree}{S}
		\EndIf
		\If{$r$ is a leaf $(k_\ell, v_\ell)$}
			\State let $U$ be the traversal-order merge of $S$ with $(k_\ell, v_\ell)$, omitting any batch element whose key is $k_\ell$
			\If{$\len{U} = 1$}
				\State \Return $r$
			\EndIf
			\State \Return \Call{rsmt\_build\_subtree}{U}
		\EndIf
		\State parse $r$ as internal node $(d, p, r_0, r_1)$
		\If{$k[0..d) \ne p$ for the first or last key $k$ of $S$}
			\State let $t < d$ be the lowest position at which such a key diverges from $p$
			\State partition $S$ into $S_0$ and $S_1$ according to bit $t$
			\State $b \gets p[t]$ \Comment{the branch taken by the old subtree}
			\If{$b = 0$}
				\State $r_0' \gets \Call{rsmt\_merge\_batch}{r, S_0}$
				\State $r_1' \gets \Call{rsmt\_merge\_batch}{\bot, S_1}$
			\Else
				\State $r_0' \gets \Call{rsmt\_merge\_batch}{\bot, S_0}$
				\State $r_1' \gets \Call{rsmt\_merge\_batch}{r, S_1}$
			\EndIf
			\State \Return the new node $(t, p[0..t), r_0', r_1')$ \Comment{edge split}
		\EndIf
		\State partition $S$ into $S_0$ and $S_1$ according to bit $d$
		\State $r_0' \gets \Call{rsmt\_merge\_batch}{r_0, S_0}$
		\State $r_1' \gets \Call{rsmt\_merge\_batch}{r_1, S_1}$
		\State \Return the node $(d, p, r_0', r_1')$
	\EndFunction
\end{algorithmic}

Because $S$ is already sorted in traversal order, the implementation can locate the first relevant divergence by inspecting only the first and the last key of the slice. Thus each recursive call either descends into the already committed frontier or constructs a fresh subtree for a non-empty batch slice, without extra per-key membership lookups.

The helper procedure $\textsc{rsmt\_build\_subtree}(S)$ is applied only to a non-empty traversal-sorted slice:

\begin{enumerate}
	\item If $\len{S} = 1$, create a leaf for the single pair in $S$ and return it.
	\item Otherwise compute the first bit position $d$ at which the first and the last key of $S$ differ. This determines the unique bifurcation depth of the subtree root.
	\item Partition $S$ into $S_0$ and $S_1$ according to bit $d$, recursively build the two child subtrees, and return the internal node $(d, p, \cdot, \cdot)$ whose region $p = k[0..d)$ is the shared prefix of the keys $k$ in $S$.
\end{enumerate}

Unlike the consistency-proof-producing batch insertion path, this proof-free variant does not return the subset of leaves that were actually inserted. That auxiliary list is only needed for pairing with a consistency proof.

For an unordered batch of size $m$, the initial sort costs $O(m \log m)$. The recursive merge then touches only the affected search frontier of the existing tree and the newly created nodes, so the total running time remains within the same $O(m \log m)$ bound.


\subsection{Shard Split}\label{sec:rsmt-shard-split}

When a shard $\sigma$ is split into two sub-shards $\sigma \| 0$ and $\sigma \| 1$ (Sec.~\ref{sec:sharding}), the RSMT maintained by the parent shard must be partitioned accordingly: each successor keeps only those leaves whose keys route to it under the sharding function $f_{\mathcal{SH}}$. In the bit-numbering convention of Section~\ref{app:rsmt}, a key $k$ belongs to $\sigma \| b$ if bit $|\sigma|$ of $k$ equals $b$.

\subsubsection{Function $\textsc{rsmt\_shard\_split}$}\label{sec:rsmt-shard-split-fn}

Retains the sub-shard assigned by the bit $b$ and discards the complement.

\textbf{Input}:
\begin{enumerate}
	\item $T$ -- RSMT of the current shard $\sigma$ (every leaf key agrees with $\sigma$ on bits $0, \ldots, |\sigma|-1$)
	\item $s \in \{0, \ldots, 255\}$ -- split depth, equal to $|\sigma|$
	\item $b \in \{0, 1\}$ -- selector bit of the sub-shard to retain
\end{enumerate}

\textbf{Output}: $T'$ -- RSMT with exactly those leaves of $T$ whose bit $s$ equals $b$

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_shard\_split}{$T, s, b$}
		\If{$T.\mathsf{root} = \bot$}
			\State \Return empty tree
		\EndIf
		\State $T'.\mathsf{root} \gets \Call{rsmt\_split\_node}{T.\mathsf{root}, s, b}$
		\State \Return $T'$
	\EndFunction
\end{algorithmic}

The helper $\textsc{rsmt\_split\_node}(r, s, b)$ descends the subtree rooted at $r$ to the first node whose bifurcation depth is at least $s$:

\begin{algorithmic}
	\Function{rsmt\_split\_node}{$r, s, b$}
		\If{$r$ is a leaf $(k, v)$}
			\If{bit $s$ of $k$ equals $b$} \Return $r$
			\Else
				\State \Return $\bot$
			\EndIf
		\EndIf
		\State parse $r$ as internal node $(d, p, c_0, c_1)$
		\If{$d = s$}
			\State \Return $c_b$ \Comment{selector bit resolves the split}
		\Else \Comment{$d > s$: all descendant keys agree on bit $s$}
			\State $b^* \gets p[s]$ \Comment{bit $s$ of the region, equal to bit $s$ of any descendant key}
			\If{$b^* = b$}
				\State \Return $r$
			\Else
				\State \Return $\bot$
			\EndIf
		\EndIf
	\EndFunction
\end{algorithmic}

\textbf{Properties}.
\begin{itemize}
	\item \emph{No rehashing}: Because internal-node hashes commit to the absolute bifurcation depth $d$ and region $p$ (Sec.~\ref{sec:rsmt-node-hash}) and leaf hashes commit to the full key (Sec.~\ref{sec:rsmt-leaf-hash}), every retained node keeps its pre-split hash. In particular, the regions of retained nodes keep their full absolute prefixes, including the bits fixed by the shard identifier. The post-split root hash equals the hash of whichever existing node the procedure returns.
	\item \emph{Inclusion certificate change}: An inclusion certificate (Sec.~\ref{sec:rsmt-inclusion-cert}) issued against the post-split root differs from a pre-split one only in that the entry at depth $s$, if any, is dropped from both the bitmap and the sibling list. This is compensated by the Unicity Certificates being larger, due to the extra entry in the Shard Tree.
	\item The procedure may be executed as many times as necessary to obtain the necessary number of shards.
\end{itemize}

\subsection{Inclusion Certificate}\label{sec:rsmt-inclusion-cert}

An \emph{Inclusion Certificate}\index{Inclusion Certificate!RSMT} $C^\mathsf{inc}$ demonstrates that a specific $(k, v)$ pair exists in a tree with root hash $\rho$.

\subsubsection{Format}

An inclusion certificate is a pair $C^\mathsf{inc} = (\mathit{bitmap}, \mathit{siblings})$ where:
\begin{itemize}
	\item $\mathit{bitmap} \in \bytes{32}$ -- a 256-bit vector. Bit $d$ is stored as bit $d\bmod 8$ of byte $\lfloor d/8\rfloor$ and is set if and only if a sibling hash is required at depth $d$ on the path from the leaf to the root.
	\item $\mathit{siblings} = \langle s_1, \ldots, s_n \rangle \in \hashtype^n$ where $n = \mathsf{bitcount}(\mathit{bitmap})$ -- sibling hashes ordered root-to-leaf (ascending depth).
\end{itemize}

Junction regions are not part of the certificate: the verifier reconstructs the expected region at each depth $d$ on the path as $k[0..d)$ from the queried key itself.

\noindent\textbf{Wire format}: $\mathit{bitmap}[32] \,\|\, s_1[32] \,\|\, \ldots \,\|\, s_n[32]$, for a total of $32 + 32n$ bytes.

\subsubsection{Generation}

To generate an inclusion certificate for key $k$, walk from the root to the leaf matching $k$. At each internal node with bifurcation depth $d$ on the path, set bit $d$ in the bitmap and append the sibling subtree's hash to the siblings list.

\subsubsection{Function $\textsc{rsmt\_verify\_inclusion}$}\label{sec:rsmt-verify-inclusion}

Verifies an inclusion certificate by reconstructing the root hash from the leaf upward.

\textbf{Input}:
\begin{enumerate}
	\item $C^\mathsf{inc} = (\mathit{bitmap}, \mathit{siblings})$ -- inclusion certificate
	\item $\rho \in \hashtype$ -- expected (authentic) root hash
	\item $k \in \bytes{32}$ -- key
	\item $v \in \bytestr$ -- value
\end{enumerate}

\textbf{Output}: \TRUE or \FALSE

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_verify\_inclusion}{$C^\mathsf{inc}, \rho, k, v$}
		\State $h \gets \Call{rsmt\_leaf\_hash}{k, v}$
		\State $j \gets \len{\mathit{siblings}}$
		\For{$d \gets 255$ \textbf{downto} $0$}
			\If{bit $d$ of $\mathit{bitmap}$ is not set}
				\State \textbf{continue}
			\EndIf
			\State $j \gets j - 1$
			\If{$j < 0$} \Return \FALSE \EndIf
			\State $s \gets \mathit{siblings}[j]$
			\State $p \gets k[0..d)$ \Comment{Expected region, derived from the key}
			\If{bit $d$ of $k$ is $1$} \Comment{Leaf went right at depth $d$}
				\State $h \gets \Call{rsmt\_node\_hash}{s, h, d, p}$
			\Else
				\State $h \gets \Call{rsmt\_node\_hash}{h, s, d, p}$
			\EndIf
		\EndFor
		\State \Return $(j = 0) \land (h = \rho)$
	\EndFunction
\end{algorithmic}

\subsection{Non-Inclusion Certificate}\label{sec:rsmt-non-inclusion-cert}

A \emph{Non-Inclusion Certificate}\index{Non-Inclusion Certificate!RSMT} $C^\mathsf{exc}$ demonstrates that a key $k$ does \emph{not} appear in a tree with root hash $\rho$. It is a chain of one-level node openings along the key-directed descent from the root. In a path-compressed RSMT, the descent for an absent key terminates either at a junction whose region is not a prefix of $k$ (the key diverges inside a compressed edge), or at a leaf carrying a different key. Opening every node on the descent binds the terminal to the root: the hash of each opening must equal the child digest expected by the previous opening.

\subsubsection{Format}

A non-inclusion certificate is a sequence $C^\mathsf{exc} = \langle e_1, \ldots, e_\ell \rangle$ of openings, ordered from the root downward, where each element is either
\begin{itemize}
	\item a \emph{junction opening} $(d, p, h_L, h_R)$ with $d \in \{0, \ldots, 255\}$, $p \in \bits{d}$, $h_L, h_R \in \hashtype$; or
	\item a \emph{leaf opening} $(k_\ell, v_\ell)$ with $k_\ell \in \bytes{32}$, $v_\ell \in \bytestr$; only admissible as the last element.
\end{itemize}

\noindent\textbf{Wire format}: junction opening $\mathtt{0x01} \,\|\, d[1] \,\|\, \langle p \rangle[32] \,\|\, h_L[32] \,\|\, h_R[32]$ (98 bytes); leaf opening $\mathtt{0x00} \,\|\, k_\ell[32] \,\|\, v_\ell$ ($33 + \len{v_\ell}$ bytes).

For the special case of an empty tree ($\rho = \bot$), the non-inclusion certificate is the empty sequence---every key is trivially absent.

\subsubsection{Generation}

To generate a non-inclusion certificate for an absent key $k$, walk from the root along the descent directed by $k$. At each junction whose region is a prefix of $k$ (i.e., $k[0..d) = p$), append the junction's opening and descend into the child selected by $k[d]$. Stop after appending the opening of the first node that terminates the descent: a junction whose region is not a prefix of $k$, or the reached leaf (its key differs from $k$, since $k$ is absent).

\subsubsection{Function $\textsc{rsmt\_verify\_non\_inclusion}$}\label{sec:rsmt-verify-non-inclusion}

Verifies that key $k$ is absent from a tree with root hash $\rho$.

\textbf{Input}:
\begin{enumerate}
	\item $C^\mathsf{exc} = \langle e_1, \ldots, e_\ell \rangle$ -- non-inclusion certificate
	\item $\rho \in \hashtype \opt$ -- root hash ($\bot$ if tree is empty)
	\item $k \in \bytes{32}$ -- queried key
\end{enumerate}

\textbf{Output}: \TRUE or \FALSE

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_verify\_non\_inclusion}{$C^\mathsf{exc}, \rho, k$}
		\If{$\rho = \bot$}
			\State \Return $C^\mathsf{exc} = \langle \rangle$ \Comment{Empty tree contains no keys}
		\EndIf
		\If{$C^\mathsf{exc} = \langle \rangle$}
			\State \Return \FALSE
		\EndIf
		\State $h \gets \rho$ \Comment{Expected digest of the next opening}
		\For{$i \gets 1$ \textbf{to} $\ell$}
			\If{$e_i$ is a leaf opening $(k_\ell, v_\ell)$}
				\State \Return $(i = \ell) \land (\Call{rsmt\_leaf\_hash}{k_\ell, v_\ell} = h) \land (k_\ell \ne k)$
			\EndIf
			\State parse $e_i$ as junction opening $(d, p, h_L, h_R)$
			\If{$\Call{rsmt\_node\_hash}{h_L, h_R, d, p} \ne h$}
				\State \Return \FALSE
			\EndIf
			\If{$k[0..d) \ne p$}
				\State \Return $i = \ell$ \Comment{Divergence junction: valid terminal}
			\EndIf
			\State $h \gets h_R$ \textbf{if} $k[d] = 1$ \textbf{else} $h_L$ \Comment{Key-directed descent}
		\EndFor
		\State \Return \FALSE \Comment{Chain ended without a terminal}
	\EndFunction
\end{algorithmic}

Note that the terminal leaf opening exposes the neighbouring leaf's value $v_\ell$; in the Unicity SMT, values are transaction hashes, so no payload data is revealed.


\subsection{Consistency Proof}\label{sec:rsmt-consistency-proof}

A \emph{consistency proof}\index{consistency proof} demonstrates that a batch $B$ of new key-value pairs was correctly inserted into a tree with root $\rho_0$, producing a new tree with root $\rho_1$. An accepting proof establishes \emph{append-only consistency}\index{append-only consistency}, which combines two properties:
\begin{itemize}
	\item \emph{prior-state preservation}: no pre-existing leaf was removed or modified; and
	\item \emph{coherent placement}: every new leaf is placed at the position determined by its own key.
\end{itemize}


\subsubsection{Sort Order}\label{sec:rsmt-sort-order}

The batch and proof use the tree traversal order defined in Section~\ref{app:rsmt}. The sort key of a 256-bit key $k=k_0\|k_1\|\cdots\|k_{31}$ is obtained by independently reversing the bits within each byte while preserving byte order. This places bit $0$ of $k_0$ first in lexicographic order, then bit $1$ of $k_0$, and so on through bit $7$ of $k_{31}$, matching the tree's branching order.

\subsubsection{Proof Format}

The proof $\pi$ is a flat stream of opcodes emitted from a post-order traversal of the touched part of the post-state tree, in the order defined above:

\begin{center}
\begin{tabular}{clcl}
\textbf{Opcode} & \textbf{Semantics} & \textbf{Wire encoding} & \textbf{Size} \\
\hline
$\mathtt{S}(h)$ & Preserved subtree, opaque & $\mathtt{0x00} \,\|\, h[32]$ & 33 bytes \\
$\mathtt{L}$ & New leaf (consumed from sorted batch) & $\mathtt{0x01}$ & 1 byte \\
$\mathtt{N}(d)$ & Junction at bifurcation depth $d$ & $\mathtt{0x02} \,\|\, d[1]$ & 2 bytes \\
$\mathtt{O}(d, p, h_L, h_R)$ & Preserved junction, opened one level & $\mathtt{0x03} \,\|\, d[1] \,\|\, \langle p \rangle[32] \,\|\, h_L[32] \,\|\, h_R[32]$ & 98 bytes \\
$\mathtt{O_L}(k, v)$ & Preserved leaf, opened & $\mathtt{0x04} \,\|\, k[32] \,\|\, v$ & $33 + \len{v}$ bytes \\
\end{tabular}
\end{center}

\noindent In the post-order encoding, the two children of every $\mathtt{N}$ node precede it in the stream (left child first, then right child), enabling efficient validation using a stack machine algorithm.

An opaque $\mathtt{S}$ is admissible only where the parent junction already exists in the pre-state. Where a preserved subtree becomes the child of a junction created this round (an edge split, including the leaf-merge case), it must be presented opened, as $\mathtt{O}$ or $\mathtt{O_L}$, so that the verifier can check the new edge against the child's authenticated depth and region. Junction regions are not carried by $\mathtt{N}$ opcodes; the verifier derives them from the children.

\subsubsection{Generation}

To generate the proof during batch insertion, perform a post-order traversal of the tree regions affected by the insertion. Emit $\mathtt{S}(h)$ for every preserved subtree whose parent junction is pre-existing (carrying the subtree's pre-existing hash); $\mathtt{O}(d, p, h_L, h_R)$ or $\mathtt{O_L}(k, v)$ for every preserved junction or leaf that becomes a child of a junction created this round; $\mathtt{L}$ for every newly inserted leaf; and $\mathtt{N}(d)$ for every junction on the touched frontier (pre-existing or new) at bifurcation depth $d$.


\subsubsection{Function $\textsc{rsmt\_verify\_consistency}$}\label{sec:rsmt-verify-consistency}

Verifies a consistency proof using a stack machine. The stack holds triples $(h_\mathsf{pre}, h_\mathsf{post}, a)$: the subtree's pre-state hash ($\bot$ if the subtree is new), its post-state hash, and an \emph{advice tuple} $a = (\delta, \varrho)$ carrying the depth and region of the subtree's top node---$(256, k)$ for leaves---or $a = \bot$ for opaque $\mathtt{S}$ subtrees, whose top node is not disclosed. Advice is authenticated: it is recomputed by the verifier from an opened preimage ($\mathtt{O}$, $\mathtt{O_L}$), consumed from the batch ($\mathtt{L}$), or derived at a lower junction ($\mathtt{N}$).

Processing $\mathtt{N}(d)$ enforces \emph{edge coherence}: every child that carries advice $(\delta, \varrho)$ must satisfy $\delta > d$ and $\varrho[d] = \beta$, where $\beta \in \{0, 1\}$ is the child's side, and all advised children must agree on the derived junction region $p = \varrho[0..d)$; at least one child must be advised, so $p$ is always defined. If the junction is absent from the pre-state (at least one child's pre-state hash is $\bot$), \emph{both} children must carry advice: an opaque $\mathtt{S}$ may appear only under pre-existing junctions, whose edges were checked in the round that created them and are frozen by the hashes. An accepting run therefore reconstructs $\rho_0$ exactly from preserved material, reconstructs $\rho_1$ from the same material plus the batch, and validates every edge created this round.

\textbf{Input}:
\begin{enumerate}
	\item $\pi$ -- consistency proof (sequence of opcodes)
	\item $\rho_0 \in \hashtype \opt$ -- root hash before insertion ($\bot$ if tree was empty)
	\item $\rho_1 \in \hashtype \opt$ -- root hash after insertion
	\item $B = \langle (k_1, v_1), \ldots, (k_m, v_m) \rangle$ -- batch of inserted key-value pairs
\end{enumerate}

\textbf{Output}: \TRUE or \FALSE

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_verify\_consistency}{$\pi, \rho_0, \rho_1, B$}
		\If{$B = \langle\rangle$}
			\State \Return $(\rho_0 = \rho_1) \land (\pi = \langle\rangle)$
		\EndIf
		\State Sort $B$ by $\textsc{rsmt\_sort\_key}$ to obtain $B' = \langle (k'_1, v'_1), \ldots, (k'_m, v'_m) \rangle$
		\State \Assert the keys of $B'$ are strictly increasing \Comment{In particular, no repeated keys}
		\State $\mathcal{S} \gets \langle\rangle$ \Comment{Stack of $(h_\mathsf{pre}, h_\mathsf{post}, a)$ triples}
		\State $b \gets 1$ \Comment{Batch index into $B'$}
		\For{each opcode $\mathsf{op}$ in $\pi$}
			\If{$\mathsf{op} = \mathtt{S}(h)$}
				\State push $(h, h, \bot)$ onto $\mathcal{S}$
			\ElsIf{$\mathsf{op} = \mathtt{O}(d, p, h_L, h_R)$}
				\State \Assert $0 \le d < 256$ and $p \in \bits{d}$
				\State $h \gets \Call{rsmt\_node\_hash}{h_L, h_R, d, p}$
				\State push $(h, h, (d, p))$ onto $\mathcal{S}$
			\ElsIf{$\mathsf{op} = \mathtt{O_L}(k, v)$}
				\State $h \gets \Call{rsmt\_leaf\_hash}{k, v}$
				\State push $(h, h, (256, k))$ onto $\mathcal{S}$
			\ElsIf{$\mathsf{op} = \mathtt{L}$}
				\If{$b > m$} \Return \FALSE \EndIf
				\State push $(\bot, \Call{rsmt\_leaf\_hash}{k'_b, v'_b}, (256, k'_b))$ onto $\mathcal{S}$
				\State $b \gets b + 1$
			\ElsIf{$\mathsf{op} = \mathtt{N}(d)$}
				\If{$\len{\mathcal{S}} < 2 \lor d \notin \{0, \ldots, 255\}$} \Return \FALSE \EndIf
				\State pop $(r_0, r_1, a_R)$ from $\mathcal{S}$ \Comment{Right child}
				\State pop $(\ell_0, \ell_1, a_L)$ from $\mathcal{S}$ \Comment{Left child}
				\State $p \gets \bot$
				\For{$(a, \beta) \in \langle (a_L, 0), (a_R, 1) \rangle$} \Comment{Edge coherence}
					\If{$a = (\delta, \varrho) \ne \bot$}
						\State \Assert $\delta > d$ and $\varrho[d] = \beta$
						\State \Assert $p = \bot \lor p = \varrho[0..d)$
						\State $p \gets \varrho[0..d)$
					\EndIf
				\EndFor
				\State \Assert $p \ne \bot$ \Comment{At least one child advised}
				\If{$\ell_0 = \bot \lor r_0 = \bot$} \Comment{Junction absent from pre-state}
					\State \Assert $a_L \ne \bot \land a_R \ne \bot$ \Comment{No opaque child on a new edge}
				\EndIf
				\State $h_0 \gets \Call{rsmt\_resolve\_pre\_state}{\ell_0, r_0, d, p}$
				\State $h_1 \gets \Call{rsmt\_node\_hash}{\ell_1, r_1, d, p}$
				\State push $(h_0, h_1, (d, p))$ onto $\mathcal{S}$
			\Else
				\State \Return \FALSE \Comment{Unknown opcode}
			\EndIf
		\EndFor
		\If{$b \neq m + 1 \lor \len{\mathcal{S}} \neq 1$}
			\State \Return \FALSE
		\EndIf
		\State pop $(\rho'_0, \rho'_1, a)$ from $\mathcal{S}$
		\State \Return $(\rho'_0 = \rho_0) \land (\rho'_1 = \rho_1)$
	\EndFunction
\end{algorithmic}

Any failing assertion, stack underflow or malformed operand rejects the proof. The verifier is a single linear pass with memory bounded by the tree depth. For the honest prover, the stream differs from a depth-only encoding without regions only in the openings: one $\mathtt{O}$ or $\mathtt{O_L}$ per split edge; the derived regions cost nothing on the wire.

We define the helper function $\textsc{rsmt\_resolve\_pre\_state}$, which resolves the pre-insertion hash of a node. When one child is new ($\bot$), the parent node did not exist in the pre-state; only the existing child's subtree occupied that position, so its hash passes through without hashing.


\textbf{Input}:
\begin{enumerate}
	\item $h_L \in \hashtype \opt$ -- left child pre-state hash ($\bot$ if new)
	\item $h_R \in \hashtype \opt$ -- right child pre-state hash ($\bot$ if new)
	\item $d \in \uint{8}$ -- bifurcation depth
	\item $p \in \bits{d}$ -- region
\end{enumerate}

\textbf{Output}: $h \in \hashtype \opt$

\textbf{Computation}:
\begin{algorithmic}
	\Function{rsmt\_resolve\_pre\_state}{$h_L, h_R, d, p$}
		\If{$h_L = \bot \land h_R = \bot$}
			\State \Return $\bot$ \Comment{Entire subtree is new}
		\ElsIf{$h_L = \bot$}
			\State \Return $h_R$ \Comment{Only right child existed}
		\ElsIf{$h_R = \bot$}
			\State \Return $h_L$ \Comment{Only left child existed}
		\Else
			\State \Return $\Call{rsmt\_node\_hash}{h_L, h_R, d, p}$ \Comment{Both children existed}
		\EndIf
	\EndFunction
\end{algorithmic}


\subsection{ZK-Compressed Consistency Proofs}\label{app:zk-consistency-proof}

For high-throughput scenarios with large batches, the hash-based consistency proof size of $O(j \cdot d)$ (where $j$ is the batch size and $d$ is the tree depth) may become a bandwidth bottleneck. Cryptographic zero-knowledge proofs (ZK-SNARKs or ZK-STARKs) can compress the consistency proof to constant ($O(1)$) or logarithmic ($O(\log j)$) size while maintaining tight security guarantees.

\subsubsection{ZK Proof Structure}

A ZK-compressed consistency proof proves successful execution of the $\mathsf{rsmt\_verify\_consistency}$ algorithm (Appendix~\ref{sec:rsmt-verify-consistency}) with the following parameters:

\begin{itemize}
	\item \textbf{Public inputs (instance)}: Values that must match the Input Record in the Certification Request:
	\begin{itemize}
		\item $h'$ -- previous SMT root hash (matches $\mathcal{IR}.h'$)
		\item $h$ -- current SMT root hash (matches $\mathcal{IR}.h$)
%		\item $j$ -- number of leaves in batch (matches $\ell_B$ in the Certification Request)
	\end{itemize}
	\item \textbf{Private inputs (witness)}: unique inserted batch and consistency proof $(B, \Pi_{CP})$
\end{itemize}

\subsubsection{Verification}

When receiving a ZK-compressed consistency proof, the verifier must:
\begin{enumerate}
	\item Verify that the proof is proving the right statement (hash-based consistency proof verification algorithm)
	\item Verify the ZK proof, obtaining authentic public inputs $h'$, $h$
	\item Confirm these values match the claimed Input Record
\end{enumerate}


\section{Radix Sparse Merkle Sum Trees}\label{app:rsmst}

A \emph{Radix Sparse Merkle Sum Tree}\index{Radix Sparse Merkle Sum Tree} (RSMST) is a leaf-anchored, path-compressed binary Merkle sum tree over the 256-bit key space: every leaf carries a positive amount, and every internal node commits the hashes \emph{and} the sums of its two subtrees.

The use-cases (e.g., fungible token splitting) are one-shot: tree is built once and never updated.

The RSMST shares with the RSMT (Appendix~\ref{app:rsmt}) the following conventions: the 256-bit key space and byte-order-preserving LSB-in-byte bit numbering, leaf anchoring by the full key, the absolute bifurcation-depth commitment, and the canonical construction rule (every junction sits at the first bit position at which its descendant keys diverge; single-child chains are compressed away). Differences: RSMST has a single version and a single, per-leaf verification pattern: node hashes commit child sums instead of regions, the inclusion proof format is different, and none of the RSMT's version-transition logic (consistency proofs, openings, non-inclusion certificates) applies. Precise properties are given in Sec.~\ref{sec:rsmst-properties}.

\subsection{Tree Structure}

Keys are 32-byte strings. Concretely, on token splitting use-case, the key is the output token identifier $\mathsf{id}_j$, the leaf data is the 32-byte output commitment $d_j$, and the leaf amount is the positive asset allocation $v_j(\mathsf{aid})$.

An RSMST leaf carries $(k,d,v)$ where $k,d\in\bytes{32}$ and $1\leq v<2^{256}$. Notation: $d$ denotes the 32-byte leaf \emph{data} (the output commitment $d_j$ above), matching the token-split notation; bifurcation depths are always written $\delta$. An internal node carries an absolute bifurcation depth $\delta\in\{0,\ldots,255\}$ and two child pairs $(h_L,v_L)$ and $(h_R,v_R)$; its sum is $v_L+v_R$. A split allocation tree is never empty.

\textbf{Construction requirements}: the builder MUST use the canonical path-compressed trie shape defined above, MUST reject duplicate keys and zero-valued leaves, and MUST reject any internal sum overflow. Canonical shape makes the root a deterministic function of the allocation set, so any party holding the allocations can reproduce and cross-check the manifest roots. These requirements are not observable by the per-leaf verifier, and---as stated in Sec.~\ref{sec:rsmst-properties}---the verifier's security guarantees do not depend on them.

For a tree $\mathcal{S}$, $\Call{RSMSTRoot}{\mathcal{S}}$ returns the root hash and root sum. If $\mathcal{S}$ has one leaf, the root hash is that leaf hash and the root sum is that leaf amount. Otherwise, the  hash and sum of the top internal node are returned.

\subsection{Hash Computation}

Hash inputs use fixed binary encodings, not general-purpose CBOR tuples. Let $\Call{u256}{x}$ be the 32-byte big-endian encoding of $x$ and let $\Call{u8}{\delta}$ be the one-byte encoding of $\delta$. Define one-byte domain separators:
\[
\mathsf{RSMST\_LEAF}=\mathtt{0x10},\qquad
\mathsf{RSMST\_NODE}=\mathtt{0x11}.
\]

\begin{algorithmic}
	\Function{rsmst\_leaf\_hash}{$k,d,v$}
		\State \Return $\mathsf{SHA\text{-}256}(\mathsf{RSMST\_LEAF}\|k\|d\|\Call{u256}{v})$
	\EndFunction
\end{algorithmic}

\begin{algorithmic}
	\Function{rsmst\_node\_hash}{$\delta,(h_L,v_L),(h_R,v_R)$}
		\State $v \gets v_L+v_R$ using checked 256-bit addition; on overflow return $(\bot,0)$
		\State $h \gets \mathsf{SHA\text{-}256}(\mathsf{RSMST\_NODE}\|\Call{u8}{\delta}\|h_L\|\Call{u256}{v_L}\|h_R\|\Call{u256}{v_R})$
		\State \Return $(h,v)$
	\EndFunction
\end{algorithmic}

\subsection{Explicit-Depth Inclusion Proof}\label{sec:rsmst-inclusion-proof}

An RSMST inclusion proof is a leaf-to-root sequence of sibling entries:
\[
C^\mathsf{inc}=\langle(\delta_1,s_1,w_1),\ldots,(\delta_n,s_n,w_n)\rangle,
\]
where $0\leq n\leq256$, $\delta_i\in\{0,\ldots,255\}$ is the sibling's parent bifurcation depth, $s_i\in\hashtype$ is the sibling subtree hash, and $1\leq w_i<2^{256}$ is the sibling subtree sum. Depths MUST be strictly decreasing in proof order because the proof is ordered from the leaf toward the root. No bitmap is carried; the depth in each sibling entry fully identifies the branch step.

\subsection{Function $\textsc{rsmst\_verify\_inclusion}$}\label{sec:rsmst-verify-inclusion}

\textbf{Input}:
\begin{enumerate}
	\item $k \in \bytes{32}$ -- expected leaf key;
	\item $d \in \bytes{32}$ -- expected leaf data;
	\item $v \in \mathbb{N}$, $1\leq v<2^{256}$ -- expected leaf amount;
	\item $C^\mathsf{inc}$ -- explicit-depth inclusion proof; and
	\item $r \in \bytes{32}$ -- expected root hash.
\end{enumerate}

\textbf{Output}: $(b,v_\mathsf{root}) \in \bool\times\mathbb{N}$, where $b$ is the success flag and $v_\mathsf{root}$ is the reconstructed root sum.

\begin{algorithmic}
	\Function{rsmst\_verify\_inclusion}{$k,d,v,C^\mathsf{inc},r$}
		\If{$v=0$, $v\geq2^{256}$, $\len{C^\mathsf{inc}}>256$, or any field is non-canonical} \State \Return $(\FALSE,0)$ \EndIf
		\State $h \gets \Call{rsmst\_leaf\_hash}{k,d,v}$
		\State $v_\mathsf{sum} \gets v$
		\State $\delta_\mathsf{prev}\gets256$
		\ForAll{$(\delta,s,w)$ in $C^\mathsf{inc}$}
			\If{$\delta\geq\delta_\mathsf{prev}$, $\delta\notin\{0,\ldots,255\}$, $w=0$, or $w\geq2^{256}$}
				\State \Return $(\FALSE,0)$
			\EndIf
			\If{bit $\delta$ of $k$ is $0$}
				\State $(h',v')\gets\Call{rsmst\_node\_hash}{\delta,(h,v_\mathsf{sum}),(s,w)}$
			\Else
				\State $(h',v')\gets\Call{rsmst\_node\_hash}{\delta,(s,w),(h,v_\mathsf{sum})}$
			\EndIf
			\If{$h'=\bot$} \State \Return $(\FALSE,0)$ \EndIf
			\State $h\gets h'$
			\State $v_\mathsf{sum}\gets v'$
			\State $\delta_\mathsf{prev}\gets\delta$
		\EndFor
		\State \Return $((h=r),v_\mathsf{sum})$
	\EndFunction
\end{algorithmic}

The caller checks value conservation by comparing the reconstructed root sum $v_\mathsf{root}$ to the authenticated source amount for the same asset (Sec.~\ref{sec:token-splitting}). The root hash alone is not sufficient for this: sibling sums are verification-critical proof data because they are committed by every internal node hash.

\subsection{Properties}\label{sec:rsmst-properties}

The trust setting is asymmetric: the previous token owner (the splitter) who constructed the tree is untrusted; each output recipient verifies only its own leaf against the manifest's root hash and never sees the tree. Assuming collision resistance of the hash function, $\textsc{rsmst\_verify\_inclusion}$ enforces the following even against an adversarially constructed root:

\begin{itemize}
	\item \emph{Conservation (no inflation)}: for any set of verifying proofs for pairwise distinct keys against the same root hash, the leaf amounts sum to at most the reconstructed root sum. Every internal node hash commits both children's (hash, sum) pairs, so hash equality forces each chain's running sum to equal the corresponding committed subtree sum---every committed sum is a hard limit for everything provable below it. Chains for distinct keys are forced to coincide from the root down to the first junction whose depth is a bit position where the keys differ, and there they descend into the two disjoint child limits. Strict positivity of leaf amounts and sibling sums and the overflow checks close the argument.
	\item \emph{Per-key uniqueness}: for a given root hash and key, at most one (data, amount) pair is provable. The descent direction at each junction is recomputed from the key's bits, depths are hash-committed, domain separation distinguishes leaf from node preimages, and the leaf hash commits the full key, so the verifying chain for a key is unique.
	\item \emph{Root-sum binding}: all verifying proofs against the same root hash reconstruct the same root sum. The conservation comparison against the source amount is therefore consistent across independent verifiers, and the root sum does not need to travel in the manifest.
\end{itemize}

These guarantees do not rely on the tree being canonical or duplicate-free. A malformed tree can only strand sum budget in sibling entries that correspond to no provable leaf, or waste it on duplicate keys---reducing what can be minted, not inflating it. Canonical construction is required of the splitter for determinism and reproducibility only.

\textbf{Non-goals.} No non-inclusion certificates: without region commitments, the absence of a key is not provable. No consistency proofs: the tree has exactly one version, so there is no state transition to validate. Coherent placement matters only when new leaves must be positioned relative to preserved, partially opaque pre-state.
